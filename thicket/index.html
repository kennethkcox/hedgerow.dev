The Problem

Network-edge security can't protect what happens
inside your AI application.

User prompts become model inputs. Model outputs become tool arguments. Every handoff is an attack surface — and traditional security tools can't see any of it.

"The attacker doesn't need to bypass your LLM's safety instructions. They just need to get the model to attempt something your application will carry out."

🔍

Cloud scanners inspect text

They catch patterns in prompts, but cannot stop what happens after — when your application starts executing what the model told it to do.

🌐

No visibility inside the runtime

Network-edge tools see HTTP traffic. They don't see the network call, file write, or process spawn that a compromised model output triggered.

⚠️

AI-specific attack surfaces

Model files, tool results, agent-to-agent calls, and supply chain packages are vectors that general-purpose security tools don't cover.

How It Works

Four independent enforcement layers.

An attacker who bypasses one immediately hits the next. Each layer enforces at a different point in the execution path — from content scanning to kernel-level controls.

1

Prompt & Content Scanning

Every prompt, retrieved document, tool call, and model output is scanned before it reaches the model or triggers downstream execution.

2

Runtime Enforcement

Intercepts operations at the moment of execution — network calls, file writes, process spawning — before they complete.

3

OS Kernel Backstop

Kernel-level enforcement that operates independently of the application runtime. Cannot be reversed from application code.

4

eBPF DaemonSet

Kubernetes only. Extends enforcement node-wide — to every process on the host, including containers that don't import Thicket.
