Architecture

The architecture behind the findings.

Four engines run in parallel. Each targets a different class of vulnerability. Together they cover what no single approach can.

Engine 1

CodeGuard — Taint Analysis

Constructs a call graph and control flow graph for your codebase, then traces data from sources to sinks across file and function boundaries.

Most taint engines stop at the file boundary. CodeGuard follows taint through imports, function arguments, and return values — across your entire Python codebase. For Java, JavaScript, Go, and C#, it uses CFG-based taint within files.

What it finds

  • SSRF chains that start in an HTTP parameter and terminate in an outbound HTTP call four functions later
  • Code injection where LLM output flows to eval()
  • Unsafe deserialization where a user-controlled path reaches torch.load()
  • IDOR where a route handler reaches object-level data without a permission check

Path-sensitive guards

CodeGuard detects boolean guards — if config.unsafe: return, if weights_only: ... — and adjusts confidence accordingly. A sanitizer-wrapped path gets a lower score. A raw, unguarded path gets a high one.

Engine 2

NeuroScan — Pattern Matching

Runs 125+ detection rules using tree-sitter AST parsing and regex matching against your source files.

Fast, language-agnostic, and comprehensive. No call graph required — rules fire against the AST directly. Catches hardcoded secrets, debug configurations, weak cryptography, missing security headers. CodeGuard provides depth; NeuroScan provides breadth.

Engine 3

DepGuard — Supply Chain with Reachability

Parses your requirements.txt, poetry.lock, or pyproject.toml, queries the OSV vulnerability database, then builds a call graph from your entry points to determine whether vulnerable functions are reachable.

A CVE in a package you import is only exploitable if your code calls the vulnerable function. DepGuard proves (or disproves) reachability before surfacing a finding. Unreachable CVEs are suppressed.

Entry points tracked: HTTP handlers, CLI commands, scheduled tasks, and plugin hooks. If a vulnerable function isn’t reachable from any of these, DepGuard does not report it.

Engine 4

Huntr Agent — Agentic Analysis

An 8-stage autonomous pipeline that uses LLMs to discover and confirm vulnerability hypotheses. It reasons about exploit chains and attempts to confirm exploitability.

#StageWhat happens
1ReconParse all source files; extract HTTP routes, model files, SSRF/command/LFI sinks
2DepScanSCA with call-graph reachability
3HypothesizeBatch 50 findings, run parallel LLM analysis, synthesize exploit hypotheses
4Deep-DiveRe-scan specific files and functions named in hypotheses
5ExploitConfirm exploitability via direct AST taint analysis
6WebExploitIf HTTP sinks exist, fire live probes — SSRF, path traversal, command injection, SQLi
7VariantExpandFind similar patterns in other files after initial confirmation
8ReportGenerate full write-up with attacker steps and remediation guidance

LLM backend: Huntr Agent works with Gemini, Claude, OpenAI, or MiniMax. The free Gemini API tier is sufficient for most repositories.

Full pipeline

Input to output.

Input: GitHub repo URL or local path | v File discovery (excludes node_modules, .venv, build artifacts) | v Four engines in parallel: |-- CodeGuard (inter-procedural taint analysis) |-- NeuroScan (AST pattern matching, 125+ rules) |-- DepGuard (SCA with reachability verification) +-- Huntr Agent (agentic hypothesis + exploit confirmation) | v Finding deduplication (fingerprint-based merge) | v Enrichment (confidence scoring, remediation suggestions) | v Output: JSON — SARIF — HTML — CSV — CycloneDX AI-BOM
Deployment

Run it how you want.

CLI

rowan scan ./my-project --format sarif --output findings.sarif

Run locally. No infrastructure required. Outputs to file.

Web API

FastAPI dashboard with project management, finding triage, REST endpoints, and webhook notifications. Single Docker container or distributed with Celery + Redis for large repositories.

Self-hosted cloud

Azure Cosmos DB backend for multi-instance deployments. Encrypted LLM API keys at rest. API key authentication on all endpoints.
