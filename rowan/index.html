The limitation

Pattern matching cannot follow data.

Traditional SAST tools flag dangerous function calls. They cannot determine whether the arguments are attacker-controlled. For AI and ML code, that distinction is everything.

Pattern-based SAST

Sees the call. Misses the context.

  • Cannot distinguish requests.get(user_url) from requests.get(api_url)
  • Cannot trace a filename through three function boundaries into a model loader
  • Cannot connect an SSRF response body to a downstream deserialization sink
  • Produces hundreds of findings per repository — the majority noise
Rowan

Traces the data. Reasons about exploitability.

  • Tracks attacker-controlled data from entry point to dangerous sink across file and function boundaries
  • Only surfaces findings with a confirmed taint path — no path, no finding
  • LLM reasoning operates on graph-filtered code slices, not raw source
  • Focused output — a small, dense set a reviewer can work through

Pattern-based SAST and Rowan are complementary, not competitive. Pattern tools have breadth. Rowan has depth on the findings that matter most.

Pipeline

Graph analysis, taint propagation,
then LLM reasoning.

Five stages in sequence. The LLM never sees raw source — only the code slices that survive the deterministic graph and taint filters.

AST & Call Graph Construction

tree-sitter builds call graphs, module dependency graphs, and entry-point enumerations across the codebase. Attacker-reachable surfaces — HTTP parameters, CLI arguments, file reads, environment variables — are identified and ranked by blast radius and privilege boundary.

Taint Propagation

Attacker-controllable data is traced through function boundaries to dangerous sinks. Only sink locations with a confirmed taint path advance. Everything else is discarded before the LLM is involved.

Pre-analysis & Ranking

Blast radius estimation, privilege boundary analysis, and entry-point exposure scoring. Candidates are ordered by likely severity before reasoning begins.

Candidate Filtering

Deterministic AST-level pre-filters remove known false-positive patterns — safe API variants, internally-trusted data flows, defensive code incorrectly flagged as a sink — before any LLM call is made.

LLM Exploit Reasoning

Each surviving candidate is evaluated by an LLM that receives only the graph-filtered code slice — not the full file, not the full repository. It produces a hypothesis, a confidence score, and an exploit description.

Coverage

Built around the AI/ML attack surface.

The AI and ML ecosystem is built on model-loading formats that are fundamentally code-execution paths — with virtually no integrity verification between download and load. Rowan is designed around this reality.

Deserialization

The dominant vulnerability class in AI/ML. Model-loading formats — torch.load, joblib, unsafe YAML loaders, and similar serialization libraries — are code-execution primitives. Rowan traces user-controlled data to deserialization sinks across supply chain, API, and checkpoint-loading paths.

Code Injection

Dynamic evaluation of user-supplied expressions, agent-generated code, and HTTP-body payloads reaching execution contexts. Covers unauthenticated and authenticated surfaces, with blast radius scored separately.

SSRF

User-controlled URLs reaching outbound HTTP calls, including redirect-bypass chains. Particularly relevant to RAG pipelines, model-fetching infrastructure, and embedding services that retrieve remote content.

Command Injection

Shell invocations constructed from attacker-controllable data. Rowan distinguishes genuinely injectable patterns from safe subprocess usage — a distinction pattern-based scanners cannot make reliably.

Path Traversal / LFI

User-controlled file paths reaching read, write, or execution operations. Filtered carefully to separate genuine traversal risk from the large volume of safe variable-path patterns common in ML codebases.

Multi-step exploit chains

Supply-chain paths where a user-controlled identifier triggers a download with no integrity check, then unsafe loading. SSRF-to-RCE chains. File-write-to-command-injection sequences. Single-step scanners miss these entirely.

Exploit chains

The attacks that require connecting the dots.

The most dangerous vulnerabilities in AI infrastructure are multi-hop chains that cross module boundaries, services, and trust levels. Pattern-based tools see one step at a time and miss the chain entirely.

Supply chain → unsafe load→User-controlled repository identifier→Download with no integrity check→Model loaded via unsafe deserializer→Arbitrary code execution

SSRF to RCE→User-controlled URL→Outbound HTTP request→Response body written to disk→Downstream unsafe load

File write to command injection→User-controlled path→Library writes file to attacker-chosen location→Downstream tool execution picks up that file
